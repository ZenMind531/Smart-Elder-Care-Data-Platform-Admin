# 后端接口约束与接口文档

## 一、文档目的

本文件用于统一后端开发规范，避免多人开发时出现接口格式不一致、字段名称不一致、路径冲突、代码结构混乱、合并困难等问题。

所有后端成员开发前必须先阅读本文件，并按照本文档中的接口路径、返回格式、字段命名和 Git 规范进行开发。

## 二、人员分工对应模块

| 成员 | 负责范围 | 主要模块 |
|---|---|---|
| 成员 A | 项目负责人 / 前端 | 前端页面、接口对接、数据看板、项目合并协调 |
| 成员 B | 后端基础模块 | 登录权限、用户管理、角色管理、老人账户、医生账户 |
| 成员 C | 老人健康业务模块 | 老人档案、健康数据、健康预警 |
| 成员 D | 报告与设备业务模块 | 评估报告、重点人群、设备管理 |

## 三、后端统一技术约束

### 1. 技术栈

- Java 17 或 Java 8，组内必须统一版本。
- Spring Boot。
- MyBatis-Plus。
- MySQL 8.0。
- Maven。
- Lombok 可使用，但所有人必须统一安装插件。
- 接口测试工具统一使用 Apifox 或 Postman。

### 2. 项目包结构

建议统一使用以下包结构，不要每个人自己新建一套风格。

```text
com.smarteldercare
├── SmartElderCareApplication.java
├── common
│   ├── result
│   ├── exception
│   ├── config
│   └── utils
├── modules
│   ├── system
│   │   ├── controller
│   │   ├── service
│   │   ├── service.impl
│   │   ├── mapper
│   │   ├── entity
│   │   ├── dto
│   │   └── vo
│   ├── elderly
│   ├── health
│   ├── report
│   ├── device
│   └── dashboard
```

### 3. 代码分层约束

- `controller`：只处理接口请求和参数接收，不写复杂业务逻辑。
- `service`：写业务逻辑。
- `mapper`：只负责数据库访问。
- `entity`：对应数据库表。
- `dto`：前端传给后端的请求参数。
- `vo`：后端返回给前端的数据对象。
- 不允许在 Controller 里直接调用 Mapper。
- 不允许一个人随意修改其他成员负责模块的核心代码。

## 四、Git 协作约束

### 1. 分支建议

```text
main                 稳定版本
dev                  日常开发合并分支
feature/frontend     前端负责人分支
feature/system       成员 B 分支
feature/elderly      成员 C 分支

```

### 2. 提交要求

每次提交信息要写清楚做了什么。

示例：

```text

feat: add elderly profile CRUD
fix: fix health warning status update
docs: update api document
```

### 3. 合并要求

- 每个人只在自己的 `feature` 分支开发。
- 合并到 `dev` 前，必须保证项目能正常启动。
- 合并前必须先 `pull` 最新 `dev` 分支。
- 不要直接提交到 `main`。
- 不要随意删除、改名公共类，例如统一返回类、异常类、配置类。
- 数据库字段有变动时，必须同步更新 SQL 文件和接口文档。
- 如果接口字段发生变化，必须提前告诉前端负责人。

## 五、接口统一规范

### 1. 基础路径

所有接口统一以 `/api` 开头。

示例：

```text
/api/auth/login
/api/elderly
/api/health-warnings
```

### 2. 请求方式

| 操作 | 请求方式 | 示例 |
|---|---|---|
| 查询列表 | GET | `GET /api/elderly` |
| 查询详情 | GET | `GET /api/elderly/{id}` |
| 新增 | POST | `POST /api/elderly` |
| 修改 | PUT | `PUT /api/elderly/{id}` |
| 删除 | DELETE | `DELETE /api/elderly/{id}` |
| 修改状态 | PATCH | `PATCH /api/health-warnings/{id}/status` |

### 3. 字段命名

- 前后端 JSON 字段统一使用小驼峰：`elderlyName`、`phoneNumber`。
- 数据库字段统一使用下划线：`elderly_name`、`phone_number`。
- 主键统一使用 `id`。
- 时间字段统一使用：`createTime`、`updateTime`。
- 日期时间格式统一：`yyyy-MM-dd HH:mm:ss`。
- 日期格式统一：`yyyy-MM-dd`。

### 4. 统一返回格式

所有接口必须返回统一格式，禁止每个接口返回不同结构。

```json
{
  "code": 200,
  "message": "success",
  "data": {},
  "timestamp": "2026-07-07 10:00:00"
}
```

### 5. 分页返回格式

分页接口统一返回以下格式。

```json
{
  "code": 200,
  "message": "success",
  "data": {
    "records": [],
    "total": 100,
    "page": 1,
    "size": 10
  },
  "timestamp": "2026-07-07 10:00:00"
}
```

### 6. 常用状态码

| code | 说明 |
|---:|---|
| 200 | 请求成功 |
| 400 | 参数错误 |
| 401 | 未登录或 token 失效 |
| 403 | 无权限 |
| 404 | 数据不存在 |
| 500 | 服务器内部错误 |

## 六、公共数据约束

### 1. 通用字段

主要业务表建议都包含以下字段：

| 字段 | 类型 | 说明 |
|---|---|---|
| `id` | BIGINT | 主键 |
| `create_time` | DATETIME | 创建时间 |
| `update_time` | DATETIME | 更新时间 |
| `deleted` | TINYINT | 逻辑删除，0 未删除，1 已删除 |

### 2. 性别字段

| 值 | 说明 |
|---|---|
| `male` | 男 |
| `female` | 女 |
| `unknown` | 未知 |

### 3. 启用状态字段

| 值 | 说明 |
|---|---|
| `enabled` | 启用 |
| `disabled` | 禁用 |

### 4. 健康预警等级

| 值 | 说明 |
|---|---|
| `low` | 低风险 |
| `medium` | 中风险 |
| `high` | 高风险 |

### 5. 健康预警处理状态

| 值 | 说明 |
|---|---|
| `pending` | 未处理 |
| `processing` | 处理中 |
| `resolved` | 已处理 |

### 6. 设备状态

| 值 | 说明 |
|---|---|
| `normal` | 正常 |
| `abnormal` | 异常 |
| `disabled` | 停用 |

## 6. 健康数据模块

负责人：成员 C

### 6.1 健康数据列表

```text
GET /api/health-records?page=1&size=10&elderlyId=1
```

### 6.2 新增健康数据

```text
POST /api/health-records
```

请求参数：

```json
{
  "elderlyId": 1,
  "elderlyName": "王建国",
  "retestWarningId": null,
  "systolicPressure": 145,
  "diastolicPressure": 92,
  "bloodSugar": 7.8,
  "heartRate": 88,
  "temperature": 36.7,
  "recordTime": "2026-07-07 10:00:00",
  "remark": "血压偏高"
}
```

说明：

- `elderlyId` 优先；如果不传 `elderlyId`，可以传 `elderlyName`。
- 只传 `elderlyName` 时，后端按姓名查老人；查不到会新建基础老人档案；查到多个同名会提示选择具体老人 ID。
- 普通新增健康数据时不传 `retestWarningId`。
- 复测某条预警时传 `retestWarningId`，后端会把本次健康记录作为复测记录关联到该预警。
- 异常健康数据会自动生成健康预警，并在预警里保存触发预警的 `healthRecordId`。

复测请求示例：

```json
{
  "elderlyId": 1,
  "retestWarningId": 8,
  "systolicPressure": 120,
  "diastolicPressure": 78,
  "recordTime": "2026-07-07 11:00:00",
  "remark": "复测血压正常"
}
```

### 6.3 健康数据详情

```text
GET /api/health-records/{id}
```

### 6.4 删除健康数据

```text
DELETE /api/health-records/{id}
```

```json
  "warningLevel": "high",
  "warningContent": "血压超过正常范围",
  "status": "pending",
  "warningTime": "2026-07-07 10:00:00"
}
```

### 7.2 预警详情

```text
GET /api/health-warnings/{id}
```

### 7.3 新增预警

```text
POST /api/health-warnings
```

请求参数：

```json
{
  "elderlyId": 1,
  "warningType": "blood_pressure",
  "warningLevel": "high",
  "warningContent": "血压超过正常范围"
}
```

### 7.4 修改预警处理状态

```text
PATCH /api/health-warnings/{id}/status
```

请求参数：

```json
{
  "status": "resolved",
  "handleResult": "已电话联系老人并通知医生跟进"
}
```

说明：

- `healthRecordId`：触发该预警的原始健康记录 ID。
- `retestRecordId`：完成复测后关联的复测健康记录 ID。
- 复测推荐通过 `POST /api/health-records` 新增复测健康记录并传 `retestWarningId`，由后端自动判断是否将预警改成 `resolved`。

### 7.5 删除预警

```text
DELETE /api/health-warnings/{id}
```

## 九、前后端联调约定

### 1. 接口未完成时的处理方式

如果后端接口还没有完成，前端可以先使用 Mock 数据开发页面，但字段名必须按照本文档来，不要自己临时起字段名。

### 2. 接口变更规则

接口路径、请求字段、返回字段一旦改动，必须同步通知负责人，并更新本文档。

### 3. 联调顺序

建议按以下顺序联调：

1. 登录接口。
2. 老人档案列表。
3. 健康预警列表。
4. 设备列表。
5. 评估报告列表。
6. 首页数据看板。
7. 报表统计图表。

## 十、禁止事项

- 禁止每个模块返回不同 JSON 格式。
- 禁止一个接口同时返回字符串、对象、数组等不固定结构。
- 禁止字段名中英文混用，例如 `laorenName`。
- 禁止一个人直接修改其他成员模块的核心代码。
- 禁止直接向 `main` 分支提交未测试代码。
- 禁止删除公共配置类、统一返回类、全局异常处理类。
- 禁止随意改数据库表名和字段名。

## 十一、建议先完成的公共类

后端正式开发前，成员 B 建议先创建以下公共类，其他成员直接复用：

```text
ApiResponse.java       统一返回结果
PageResult.java        分页返回结果
ResultCode.java        状态码枚举
GlobalExceptionHandler.java 全局异常处理
BaseEntity.java        公共实体字段
```

统一返回类示例：

```java
public class ApiResponse<T> {
    private Integer code;
    private String message;
    private T data;
    private String timestamp;
}
```

分页返回类示例：

```java
public class PageResult<T> {
    private List<T> records;
    private Long total;
    private Long page;
    private Long size;
}
```

## 十二、总结

后端开发时最重要的是统一：

- 统一接口路径。
- 统一请求字段。
- 统一返回格式。
- 统一数据库命名。
- 统一代码结构。
- 统一 Git 合并流程。

只要大家按照这份文档开发，最后合并和前后端联调会顺利很多。
