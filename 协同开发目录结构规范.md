# 协同开发目录结构规范

## 一、文档目的

本文件用于约定智慧医养后台管理系统多人协作时的项目目录结构，避免每个成员各建一套目录、接口和代码风格不统一，导致后期合并困难。

所有成员应在同一个 Spring Boot 项目骨架下开发，只在自己负责的模块目录中提交代码。

## 二、后端推荐总目录

```text
src/main/java/com/smarteldercare
```

## 三、公共目录说明

### 1. `common`

`common` 目录用于存放全项目通用代码，建议由成员 B 或项目负责人统一维护。

常见内容：

- `ApiResponse.java`：统一接口返回结构
- `PageResult.java`：分页返回结构
- `ResultCode.java`：状态码枚举
- `GlobalExceptionHandler.java`：全局异常处理
- `BaseEntity.java`：公共实体字段，例如 `id`、`createTime`、`updateTime`、`deleted`

其他成员需要使用公共类时，优先复用已有代码，不要重复创建一套类似的返回类、分页类或异常处理类。

### 2. `modules`

`modules` 目录用于存放各业务模块，每个成员只在自己负责的模块中开发。

| 目录 | 负责人 | 说明 |
|---|---|---|
| `system` | 成员 B | 登录、用户、角色、医生账户、权限 |
| `elderly` | 成员 C | 老人档案管理 |
| `health` | 成员 C | 健康数据、健康预警 |
| `report` | 成员 D | 评估报告、重点人群 |
| `device` | 成员 D | 设备管理 |
| `dashboard` | 成员 A 配合全员 | 首页统计、图表数据接口 |

## 五、每层代码职责

| 层级 | 说明 |
|---|---|
| `controller` | 接收请求、校验基础参数、返回统一响应，不写复杂业务逻辑 |
| `service` | 定义业务接口 |
| `service.impl` | 编写具体业务逻辑 |
| `mapper` | 数据库访问层，配合 MyBatis-Plus 使用 |
| `entity` | 数据库表对应实体类 |
| `dto` | 前端传给后端的请求参数对象 |
| `vo` | 后端返回给前端的数据对象 |

禁止在 `controller` 中直接调用 `mapper`。

## 六、SQL 文件目录建议

项目根目录下建议创建 `sql` 文件夹，统一存放数据库脚本。

```text
sql
├── schema.sql
├── data.sql
├── member-b-schema.sql
├── member-b-data.sql
├── member-c-schema.sql
├── member-c-data.sql
├── member-d-schema.sql
└── member-d-data.sql
```

成员 C 的 SQL 可以对应：

```text
sql/member-c-schema.sql
sql/member-c-data.sql
```

如果暂时没有统一 `sql` 目录，也可以先放在项目根目录，后续由负责人统一整理。

## 七、Git 分支建议

```text
main                  稳定版本
dev                   日常开发合并分支
feature/frontend      成员 A 前端分支
feature/system        成员 B 后端基础分支
feature/elderly       成员 C 老人健康业务分支
feature/report-device 成员 D 报告与设备分支
```

成员 C 建议在 `feature/elderly` 分支开发。

## 八、成员 C 不建议修改的内容

成员 C 开发时，除非团队明确约定，否则不要直接修改：

- `modules/system`：登录、用户、角色、权限模块
- `modules/report`：评估报告模块
- `modules/device`：设备模块
- `common` 下的公共返回类、异常处理类、配置类
- 其他成员已经负责的核心业务代码

如果确实需要修改公共代码，应先和负责人或对应成员沟通。

## 九、接口与字段命名规则

### 1. 接口路径

所有接口统一以 `/api` 开头。

成员 C 负责接口：

```text
/api/elderly
/api/health-records
/api/health-warnings
```

### 2. JSON 字段

前后端 JSON 字段统一使用小驼峰：

```text
elderlyName
phoneNumber
riskLevel
warningLevel
recordTime
```

### 3. 数据库字段

数据库字段统一使用下划线：

```text
elderly_name
phone_number
risk_level
warning_level
record_time
```

## 十、协同开发注意事项

- 不要每个成员自己创建一套 `common`、`controller`、`service` 结构。
- 不要修改别人模块的接口路径和字段名。
- 数据库字段变化后，必须同步更新 SQL 文件和接口文档。
- 接口返回结构必须统一使用 `ApiResponse`。
- 分页接口必须统一使用 `PageResult`。
- 每次提交前确认项目能正常启动。
- 合并到 `dev` 前，先拉取最新代码并解决冲突。

## 十一、成员 C 最低完成范围

成员 C 至少需要完成：

- 老人档案增删改查
- 健康数据新增和查询
- 健康预警查询
- 健康预警处理状态修改
- 老人档案和健康预警数据能支持首页看板统计

优先级建议：

1. 先完成 `elderly_profile` 表和老人档案接口。
2. 再完成 `health_record` 表和健康数据接口。
3. 最后完成 `health_warning` 表和健康预警接口。
4. 有时间再实现新增健康数据后自动生成预警。
