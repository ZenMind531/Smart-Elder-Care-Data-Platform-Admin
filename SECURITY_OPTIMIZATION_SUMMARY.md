# 智慧养老数据平台安全优化总结

当前项目已经达到“能运行、能展示、能联调”的阶段，但安全设计仍比较薄弱。尤其是为了演示顺畅，系统中存在接口不可用时进入演示模式、前端缓存用户角色、登录态校验不够严格等问题。这些机制适合本地开发和答辩展示，但不适合作为真实生产系统的安全方案。

因此，安全优化建议按优先级分为五层推进：先挡住明显风险，再完善权限闭环，最后补充审计、接口安全和生产部署安全。

## 一、第一优先级：登录和会话安全

登录和会话安全是当前最需要优先处理的部分。现在最主要的风险包括：

- 后端接口不可用时，前端可以自动进入演示模式。
- token 和 user 信息存储在 localStorage 中。
- 用户角色信息由前端缓存，存在被手动篡改的风险。
- 路由守卫更多关注角色访问控制，但没有严格做到“未登录禁止进入后台”。

### 1. 生产环境关闭演示模式

演示模式只能用于本地开发、答辩展示或接口异常时的临时演示，生产环境必须禁止。

建议通过环境变量控制：

```env
VITE_ENABLE_DEMO_LOGIN=false
```

只有在本地开发环境中显式开启该变量时，才允许 demo 登录。生产环境中即使后端接口不可用，也不应该自动进入后台。

### 2. 没有 token 不允许进入后台

除了登录页和注册页之外，其它后台页面都必须要求用户已登录。

建议路由守卫规则：

- `/signin`、`/signup` 可以匿名访问。
- 其它页面如果没有 token，统一跳转到 `/signin`。
- 跳转时可以保留 redirect 参数，登录成功后返回原页面。

### 3. 不信任 localStorage 中的角色

前端存储的 roleName、userInfo 只能用于界面展示，不能作为真实权限判断依据。

原因是用户可以通过浏览器开发者工具手动修改 localStorage，例如把自己从“医生”改成“系统管理员”。

正确做法是：

- 前端只把本地角色信息作为展示缓存。
- 后端必须根据 token 判断当前用户身份和权限。
- 所有关键接口都必须由后端做权限校验。

### 4. 登录后调用 `/auth/me` 校验身份

登录接口返回 token 后，前端应继续调用 `/auth/me` 获取当前用户的真实身份信息。

刷新页面时，也应该通过 `/auth/me` 恢复用户信息，而不是完全相信本地缓存。

推荐流程：

1. 用户提交账号密码。
2. 登录接口返回 token。
3. 前端保存 token。
4. 前端调用 `/auth/me`。
5. 根据 `/auth/me` 返回的 user、role、permissions 初始化登录状态。

### 5. token 过期自动退出

当前系统在接口返回 401 时会清理 token，但页面层面的反馈还不够完整。

建议统一处理：

- 所有接口返回 401 时，清除 token 和用户信息。
- 自动跳转到 `/signin`。
- 给出提示：`登录已过期，请重新登录`。

## 二、第二优先级：角色权限安全

当前前端已经做了角色菜单过滤和路由过滤，但这只能算“用户体验层权限”，不能算真正安全。

真正的权限控制必须在后端完成。

### 1. 后端必须实现 RBAC

即使医生在前端看不到用户管理、设备删除、角色修改等按钮，也仍然可以通过浏览器控制台或接口工具直接请求接口。

例如医生不应该能够调用：

```http
POST /api/users
DELETE /api/devices/{id}
PUT /api/roles/{id}
```

这些限制必须由后端根据 token 中的用户身份、角色和权限进行校验。

### 2. 权限不要只按角色名写死

当前前端权限逻辑主要类似于：

- 医生：`/health`、`/alerts`、`/elderly`
- 设备管理员：`/devices`、`/alerts`
- 系统管理员：所有模块

这种方式适合早期演示，但后续扩展性较弱。

更推荐后端返回权限码：

```json
{
  "permissions": [
    "elderly:read",
    "health:read",
    "health:create",
    "warning:handle"
  ]
}
```

前端根据权限码控制菜单、路由和按钮展示；后端根据权限码决定接口是否允许访问。

### 3. 增加按钮级权限

不同角色不仅页面不同，页面内部的操作权限也应该不同。

医生：

- 可以查看健康监测。
- 可以处理健康告警。
- 不应该删除老人档案。
- 不应该修改设备状态。

护理管理员：

- 可以维护老人档案。
- 可以填写护理记录。
- 可以处理普通告警。
- 不应该管理角色权限。

设备管理员：

- 可以维护设备。
- 可以处理设备离线告警。
- 不应该查看详细健康病史。

系统管理员：

- 可以管理账号和角色。
- 不一定需要直接操作健康记录和护理记录。

## 三、第三优先级：接口和数据安全

智慧养老系统涉及老人基础信息、健康数据、紧急联系人等内容，真实系统中属于敏感数据，不能随意暴露。

### 1. 敏感字段脱敏

页面展示手机号、身份证号、紧急联系人电话时，建议默认脱敏。

示例：

```text
手机号：138****5261
身份证号：500***************18
```

只有具备对应权限的用户才能查看完整信息。

### 2. 接口参数校验

前端可以做基础表单校验，但后端必须做最终校验。

例如：

- 年龄不能小于 0。
- 血压不能是负数。
- 电话格式必须合法。
- 身份证格式必须合法。
- roleId 不能随意传入。
- warning status 只能是指定枚举值。

### 3. 防止越权访问

不能只判断“用户是否登录”，还要判断“用户是否有权限访问这个对象”。

例如：

```http
GET /api/elderly/1
```

后端不仅要判断当前用户是否登录，还要判断当前医生、护理人员或管理员是否有权限查看这位老人的数据。

### 4. 删除操作必须二次确认

老人档案、设备、角色、健康记录等关键数据不能点一下就删除。

建议：

- 前端增加确认弹窗。
- 高风险删除操作可以要求二次确认。
- 后端记录删除审计日志。

### 5. 关键操作写审计日志

至少以下操作需要记录审计日志：

- 登录 / 登出。
- 新增老人档案。
- 修改健康数据。
- 处理告警。
- 修改角色权限。
- 删除记录。
- 设备状态变更。

审计字段建议包括：

- 操作人。
- 操作角色。
- 操作类型。
- 目标对象。
- 操作前数据。
- 操作后数据。
- IP 地址。
- 操作时间。

## 四、第四优先级：前端安全细节

这些优化主要在前端即可推进，适合当前阶段快速落地。

### 1. 统一 auth store

建议新增：

```text
src/stores/auth.ts
```

统一管理：

- token。
- userInfo。
- role。
- permissions。
- login。
- logout。
- refreshCurrentUser。

这样可以避免登录状态分散在页面、http 请求封装和路由守卫中，后续扩展也更清晰。

### 2. 生产环境禁用 console 和 demo token

当前项目中存在 demo-token 这类演示逻辑，生产环境不能出现。

建议：

- 生产环境禁止 demo 登录。
- 生产环境不生成 demo-token。
- 生产构建中减少 console 输出。

### 3. 接口 baseURL 不要写死生产地址

当前 fallback 地址类似：

```text
https://smart-eldercare.online/api
```

更稳妥的做法是必须通过 `.env` 配置接口地址，避免开发环境误请求生产接口。

示例：

```env
VITE_API_BASE_URL=https://example.com/api
```

### 4. 避免把详细错误直接暴露给用户

后端返回的错误中可能包含 SQL、服务器路径、异常堆栈等敏感信息。

前端应该统一转换成通用提示：

```text
操作失败，请稍后重试
```

详细错误只在开发环境输出到控制台，生产环境不直接展示。

### 5. 表单输入限制

注册、老人档案、健康记录等表单需要增加输入限制。

建议校验：

- 用户名长度。
- 密码长度。
- 姓名长度。
- 备注最大字数。
- 电话格式。
- 身份证格式。

### 6. 避免使用 `v-html`

目前项目中没有主动使用 `v-html` 是比较好的。Vue 默认会对文本进行转义，可以降低 XSS 风险。

后续应继续避免在不可信内容上使用 `v-html`。

## 五、第五优先级：生产部署安全

这部分主要在上线部署时处理。

### 1. HTTPS 必须稳定

生产环境必须保证：

- HTTPS 证书有效。
- Nginx 配置稳定。
- 后端服务稳定运行。
- 不能频繁出现 502、连接 reset 等问题。

### 2. Nginx 增加安全响应头

建议在 Nginx 或后端中增加安全响应头：

```nginx
add_header X-Frame-Options "DENY";
add_header X-Content-Type-Options "nosniff";
add_header Referrer-Policy "strict-origin-when-cross-origin";
add_header Permissions-Policy "camera=(), microphone=(), geolocation=()";
```

### 3. CORS 不能全放开

不要使用：

```http
Access-Control-Allow-Origin: *
```

生产环境应该只允许指定前端域名访问接口。

### 4. 接口限流

登录、注册、修改密码等接口必须增加限流，防止暴力破解。

重点接口：

- 登录接口。
- 注册接口。
- 修改密码接口。
- 找回密码接口。

### 5. 密码必须后端加密存储

后端不能明文存储密码。

建议使用：

- BCrypt。
- Argon2。

密码只保存哈希值，不能保存明文。

## 六、当前阶段最推荐先落地的 5 项

结合项目目前的阶段，最值得马上做的是：

1. 生产环境关闭 demo 登录。
2. 路由守卫增加“没有 token 必须跳转登录页”。
3. 新增 `authStore`，统一管理登录状态。
4. 角色权限从“前端写死”升级为“权限配置 + 后端校验预留”。
5. 敏感信息脱敏，包括手机号、身份证号、紧急联系人电话。

完成以上 5 项后，答辩时可以这样说明：

> 本系统不仅实现了核心功能页面，还考虑了基础安全设计，包括登录态校验、角色访问控制、生产环境禁用演示模式、敏感信息脱敏、接口 401 自动退出，并预留了后端 RBAC 权限码机制。

这个说法既贴合当前项目实际，也能体现系统从演示项目向真实业务系统演进的安全意识。

## 七、建议下一步实施方向

建议优先改造以下三个方向：

```text
src/stores/auth.ts
src/router/index.ts
src/api/http.ts
```

具体落地内容：

- 新增 authStore。
- 路由强制登录。
- 生产环境关闭 demo 登录。
- 401 自动跳转登录页。
- 当前用户通过 `/auth/me` 恢复。
- 根据角色职责继续过滤页面。
- 为后续权限码、按钮级权限和字段脱敏预留结构。

这一步属于安全地基。完成后，再继续做敏感字段脱敏、按钮级权限、审计日志和后端 RBAC，整体改造会更顺畅。
