# 前后端权限对照表

本文档用于统一智慧养老后台管理系统的前端按钮权限、页面权限和后端接口拦截规则。前端权限用于控制页面展示和减少误操作，后端权限用于真正阻止越权请求。

## 一、角色范围

当前系统只面向工作人员，不包含家属端、老人端或个人用户端。

| 角色 | 职责范围 | 默认入口 |
| --- | --- | --- |
| 系统管理员 | 系统配置、角色权限、机构资料、基础字典、设备接入和运维巡检 | `/settings` |
| 护理管理员 | 老人档案、护理记录、服务工单和告警处置 | `/` |
| 医生 | 健康监测、体征复测、健康告警和评估建议 | `/health` |

说明：旧数据或后端历史角色中的“设备管理员”，前端会统一归并为“系统管理员”。后端也建议迁移或停用“设备管理员”独立角色。

## 二、页面访问权限

| 页面 | 路由 | 可访问角色 |
| --- | --- | --- |
| 首页总览 | `/` | 系统管理员、护理管理员 |
| 老人档案 | `/elderly` | 护理管理员、医生 |
| 健康监测 | `/health` | 医生 |
| 告警中心 | `/alerts` | 系统管理员、护理管理员、医生 |
| 设备管理 | `/devices` | 系统管理员 |
| 护理记录 | `/care-records` | 护理管理员 |
| 服务预约 | `/services` | 护理管理员 |
| 系统设置 | `/settings` | 系统管理员 |
| 个人中心 | `/profile` | 系统管理员、护理管理员、医生 |
| 登录 | `/signin` | 公开页面 |
| 注册 | `/signup` | 公开页面 |

前端路由会拦截未登录访问。无 token 时跳转到：

```text
/signin?redirect=原目标地址
```

## 四、接口权限建议

后端应在接口层校验当前 token 对应用户是否具备对应权限。前端隐藏按钮不能替代后端鉴权。

| 接口 | 方法 | 建议后端权限 |
| --- | --- | --- |
| `/auth/login` | `POST` | 公开接口 |
| `/auth/register` | `POST` | 公开接口或仅系统管理员可创建，按后端实际策略决定 |
| `/auth/logout` | `POST` | 已登录 |
| `/auth/me` | `GET` | 已登录 |
| `/auth/password` | `PUT` | 已登录本人 |
| `/elderly` | `GET` | `elderly:view` |
| `/elderly/{id}` | `GET` | `elderly:view` |
| `/elderly/{id}/health-summary` | `GET` | `elderly:view` 或 `health:view` |
| `/elderly` | `POST` | `elderly:create` |
| `/elderly/{id}` | `PUT` | `elderly:update` |
| `/elderly/{id}` | `DELETE` | `elderly:delete` |
| `/health-records` | `GET` | `health:view` |
| `/health-records/{id}` | `GET` | `health:view` |
| `/health-records/trend` | `GET` | `health:view` |
| `/health-records` | `POST` | `health:create` |
| `/health-records/{id}` | `DELETE` | `health:resolve` 或更高权限 |
| `/health-warnings` | `GET` | `alerts:view` |
| `/health-warnings/{id}` | `GET` | `alerts:view` |
| `/health-warnings` | `POST` | `alerts:create` |
| `/health-warnings/{id}/status` | `PATCH` | `alerts:handle` |
| `/health-warnings/{id}` | `DELETE` | `alerts:handle` 或更高权限 |
| `/devices` | `GET` | `devices:view` |
| `/devices/{id}` | `GET` | `devices:view` |
| `/devices` | `POST` | `devices:update` |
| `/devices/{id}` | `PUT` | `devices:update` |
| `/devices/{id}/status` | `PATCH` | `devices:update` |
| `/devices/{id}` | `DELETE` | `devices:update` |
| `/roles` | `GET` | `settings:view` |

如果后端后续新增护理记录、服务预约、通知规则等接口，建议按以下规则映射：

| 业务 | 查看权限 | 新增权限 | 修改状态权限 |
| --- | --- | --- | --- |
| 护理记录 | `care:view` | `care:create` | `care:create` 或独立 `care:update` |
| 服务预约 | `services:view` | `services:create` | `services:update` |
| 通知规则 | `settings:view` | `settings:update` | `settings:update` |
| 机构资料 | `settings:view` | `settings:update` | `settings:update` |

## 五、状态码约定

| 场景 | HTTP 状态码 | 前端行为 |
| --- | --- | --- |
| 未登录、token 过期、token 无效 | `401` | 清除本地 token 和用户信息，跳转 `/signin?expired=1` |
| 已登录但权限不足 | `403` | 保持当前页面，显示“当前账号没有操作权限” |
| 请求参数错误 | `400` | 页面显示接口返回的业务错误 |
| 服务器异常 | `500` | 页面显示接口暂不可用，保留本地演示数据或提示稍后重试 |

如果后端统一返回 HTTP 200，但业务 code 表示异常，也请保持：

```json
{
  "code": 401,
  "message": "登录已过期"
}
```

或：

```json
{
  "code": 403,
  "message": "当前账号没有操作权限"
}
```

前端已经同时兼容 HTTP 状态码和业务 `code`。

## 六、隐私保护规则

前端展示层已对老人档案和告警详情中的敏感联系方式做脱敏，不影响提交给后端的数据。

| 数据 | 展示规则 |
| --- | --- |
| 手机号 | `13812345678` 展示为 `138****5678` |
| 身份证号 | `330106200001011234` 展示为 `330106********1234` |
| 空值 | 展示为 `未填写` 或 `待补充` |

后端也建议在日志、审计记录、导出文件中做脱敏，避免完整身份证号、手机号进入普通日志。

## 七、Token 存储说明

当前前端使用 `localStorage` 存储 token：

```text
smart-elder-care-token
smart-elder-care-user
```

这对课程项目和前后端联调比较方便，但生产环境建议升级为：

```text
HttpOnly Cookie + Secure + SameSite
```

原因是 `localStorage` 中的 token 可能被 XSS 脚本读取；`HttpOnly Cookie` 不能被前端 JavaScript 直接读取，风险更低。迁移时需要后端配合设置 Cookie、刷新 token 和 CSRF 策略。

## 八、前后端联调检查清单

1. 未登录访问 `/elderly`、`/settings` 等页面，应跳转登录页。
2. token 过期后请求任意业务接口，后端返回 `401`，前端应跳转 `/signin?expired=1`。
3. 医生调用 `/devices/{id}/status`，后端应返回 `403`。
4. 护理管理员调用 `/devices/{id}/status`，后端应返回 `403`。
5. 系统管理员调用 `/devices/{id}/status`，后端应允许。
6. 医生访问 `/health` 并录入体征，应允许。
7. 护理管理员访问 `/health`，前端路由应拦截到自己的默认页面。
8. 老人档案列表、告警详情中的电话和身份证应在前端展示时脱敏。
